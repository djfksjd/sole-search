# sole-search — 소상공인 지원사업 조사 스킬 설계 스펙

- **작성일**: 2026-07-20 (v2 — 외부 리뷰 반영 개정)
- **상태**: 리뷰 반영 개정본. 재리뷰 및 사용자 재승인 대기
- **형제 스킬**: [`djfksjd/ir-search`](https://github.com/djfksjd/ir-search) — 구조 계승, description 동반 수정 필요
- **장기 통합처**: KKACHI 워크스페이스 Phase 2B (소상공인·정책자금 확장)의 백엔드 스킬 후보

## 1. 목적과 포지셔닝

이미 가게·사업을 운영 중인 **소상공인**(치킨집, 카페, 미용실, 온라인셀러 등)이 자신에게 해당되는
정부·지자체 지원사업을 **정의된 공식 소스 범위 내에서 전수조사**받고, "오늘 신청할 수 있는 것"부터
마감순으로 받아보는 스킬.

"전수조사"는 무한정 커버리지 주장이 아니다: 조사 대상 소스는 본 스펙 4장에 열거된 것으로 한정하며,
모든 보고서는 어떤 소스가 성공/실패/수동확인이었는지 `coverage_manifest`로 명시한다.
조용한 누락 금지 — 커버하지 못한 영역(예: 미등록 지자체 포털)은 보고서에 한계로 적는다.

### 1.1 사용자

소상공인 본인이 직접 사용하는 것을 기본으로 한다. 인터뷰·보고서는 쉬운 말로, 관공서 용어에는
괄호 해설을 붙인다. 동시에 프로필·산출물은 정형 스키마(3.1)로 저장해 KKACHI 제품이 나중에
그대로 파싱할 수 있게 한다.

### 1.2 ir-search와의 트리거 경계 (양쪽 동시 수정)

기존 ir-search description은 "지원사업·정부지원·보조금" 의도 전반을 잡도록 넓게 정의되어 있어
sole-search만 새로 정의해서는 충돌을 피할 수 없다. **두 스킬의 description을 함께 수정**하고
다음 라우팅 규칙을 양쪽 SKILL.md에 동일하게 명문화한다:

| 신호 | 라우팅 |
|---|---|
| 소상공인, 가게, 점포, 사장님, 자영업, 소상공인24, 시설개선, 경영안정자금, 전통시장/상권 | sole-search |
| 아이템, 창업지원, 예비창업, K-Startup, TIPS, R&D 과제, 입주공간, 액셀러레이팅 | ir-search |
| 신호가 섞이거나 모호 (예: "온라인 셀러 지원금") | 한 번 질문해 분기하거나 두 조사 범위를 함께 제시 |
| 사용자 신분보다 **요청 목적** 우선 | 운영 중인 가게 사장이라도 신규 기술 아이템 R&D를 찾으면 ir-search |

ir-search 리포에 description 수정 PR을 보내는 것을 본 프로젝트의 동반 작업으로 포함한다.
evals에는 두 스킬을 함께 설치한 상태의 트리거 충돌 테스트를 넣는다 (7장).

## 2. 지원 범위

### 2.1 유형 분류 — `primary_type + tags`

지원 유형은 단일 분류가 아니라 안정 enum인 `primary_type` + 복수 `tags`로 저장한다
(희망리턴패키지처럼 교육+컨설팅+지원금이 섞인 사업 대응). 이모지·한글 라벨은 표시용으로만 쓴다.

| primary_type | 표시 라벨 | 내용 |
|---|---|---|
| `grant` | 💰 받는 돈 | 보조금·바우처 (스마트상점, 온라인판로, 마케팅, 시설개선 등) |
| `loan` | 🏦 빌리는 돈 | 정책자금 대출·보증 — "갚아야 하는 돈" 명시 |
| `advisory` | 🎓 배우고 돕기 | 교육·컨설팅·멘토링·재기지원 |

tags 예: `고용`, `판로·조달`, `전통시장·상권`, `에너지·공과금`, `재기·폐업`, `디지털전환`, `지역화폐·상권활성화`

### 2.2 포함/비포함 명시

- **포함**: 보조금·바우처, 정책자금(대출·보증), 교육·컨설팅, 고용지원(소상공인 대상 인건비 지원),
  판로·조달, 전통시장·상권 사업, 에너지·공과금 경감, 재난·손실 지원(공고형)
- **비포함(8장)**: 세무 신고 대행성 정보(세율·공제 상담), 4대보험 요율 안내 등 "공고"가 아닌 상시 제도 해설,
  신청서 작성 대행, 창업지원(ir-search), 다중 고객 관리

## 3. 워크플로

### 0단계 — 대화형 인터뷰 → `sole-profile.md`

소상공인은 참조할 프로젝트 문서가 없으므로 최초 1회 대화형 인터뷰가 입구다.
쉬운 말로, 한 번에(여러 턴으로 쪼개지 않음) 묻는다. 대화·폴더에서 이미 파악된 항목은 다시 묻지 않는다:

1. 무슨 장사/사업을 하시나요? (예: 치킨집, 네일샵, 스마트스토어) → 업종 텍스트 + 업종코드 후보 추정(신뢰도 표기)
2. 개인사업자세요, 법인이세요? 지금 정상 영업 중이신가요? (휴업/폐업 예정 포함) → entity_type, business_status
3. 가게(사업장)는 어느 동네인가요? (시·군·구까지. 본점과 사업장이 다르면 둘 다) → 지역 코드
4. 사업자 등록은 언제 하셨나요? (년·월) → registration_date (업력은 공고 기준일마다 재계산, 저장하지 않음)
5. 직원이 몇 명인가요? (4대보험 가입 기준으로, 가족·알바 포함 여부 구분) → 법령상 상시근로자 수와 단순 인원을 구분해 저장
6. 작년 매출이 대략 어느 정도예요? (구간 선택: 3천만 미만 / 3천만~1억 / 1억~5억 / 5억~10억 / 10억 이상) → 소기업·소상공인 매출액 기준 판정용
7. 대표님 나이대와 성별은? → 청년·여성 특화 사업
8. 지금 제일 필요한 건? (복수): 안 갚는 돈 / 빌리는 돈 / 가게 시설 / 온라인 판매 / 홍보 / 배우기

**소상공인 법적 판정은 "직원 수"만으로 하지 않는다**: 업종별 상시근로자 기준(5인/10인 미만) **그리고**
업종별 평균매출액 기준(소기업 요건)을 함께 본다. 매출 구간으로 판정이 갈리는 경계에 있으면
'확인 필요'로 처리하고 단정하지 않는다. 재창업·업종전환·공동대표 등 예외 신호가 보이면 프로필에 기록한다.

**개인정보 최소화**: 세금 체납, 기존 대출액, 신용점수 등 민감 정보는 인터뷰에서 묻지 않는다.
정책자금 후보가 나온 뒤 "이 요건은 해당되시는지만 확인해주세요"로 조건부 질문하고, 답의 원문 값은
프로필에 저장하지 않는다 (해당/비해당 판정 결과도 저장하지 않고 그 조사 보고서에만 반영).

#### 3.1 프로필 스키마

YAML 프론트매터는 원자 필드로 구성한다 (파생값·복합 문자열 금지). 화면용 한글 요약은 본문에 둔다:

```markdown
---
schema_version: 1
type: sole-search-profile
entity_type: individual | corporation
business_status: active | suspended | closing
industry_text: <사용자 표현 그대로>
industry_code_candidates:
  - {code: <KSIC>, label: <업종명>, confidence: high|medium|low}
registration_date: YYYY-MM
employee_count: <N>
employee_count_basis: insured | headcount
employee_count_as_of: YYYY-MM-DD
sales_band: lt_30m | 30m_100m | 100m_500m | 500m_1b | gte_1b
sales_period: <YYYY 연간>
province: <시/도>
district: <시/군/구>
hq_district: <본점 소재지, 사업장과 다를 때만>
owner_age_band: 20s|30s|40s|50s|60s_plus
owner_gender: female | male | unspecified
needs: [grant, loan, facility, online_sales, marketing, education]
last_survey_at: YYYY-MM-DD
last_survey_dir: <보고서 폴더 경로>
survey_sources: [sbiz24, bizinfo, semas_loan, ...]
---
# sole-search 프로필
<사람이 읽는 한 줄 요약>
```

재사용 시 요약을 보여주고 "바뀐 것 있나요?" 한 번만 확인한다.

### 1단계 — 수집 (소스 계약은 4장)

각 소스 어댑터는 jsonl로 출력하며 공통 필드를 갖는다:
`source, source_id(안정 ID), announce_no(공고번호, 있으면), canonical_url, title, agency, region_scope,
apply_start, apply_end, status(접수중|예산소진|상시|회차예정|마감), primary_type, tags,
attachments[{url, filename}], crawled_at, content_hash(정규화 상세본문+첨부 해시)`

정책자금(loan)은 추가로: `loan_kind(직접대출|대리대출|보증), rate, rate_as_of, limit_amount,
loan_term, grace_period, repay_method, guarantee_fee, handling_orgs[]`

**중복 제거 우선순위**: ① 공고번호 ② canonical URL ③ 시행기관+접수기간 조합 ④ (최후) 제목 유사도.

### 2단계 — 자격 판정

판정 상태는 객관 요건 중심 5단계다. "변형하면 가능" 프레이밍은 쓰지 않는다 —
실제 영업내용·등록업종이 뒷받침되지 않는 신청 유도는 허위·오인 신청 위험이 있다:

| 상태 | 의미 |
|---|---|
| `확인됨` | 공고 원문(첨부 포함)에서 모든 필수 신청자격 충족 확인 |
| `조건부` | 충족해야 할 구체 요건이 있고 합법적으로 달성 가능 (예: 교육 이수 후 신청) |
| `확인 필요` | 프로필 정보 또는 공고 원문이 부족 — 부족한 항목을 명시 |
| `신청 불가` | 필수요건 1개 이상 불충족 (근거 문구 인용) |
| `사업전환 후보` | 실제 사업전환·업종추가를 전제로만 별도 섹션에 제안 (기본 비표시) |

검증 축: 업종 제한(제외 업종), 업력(공고 기준일로 산정), 상시근로자 수, 매출액 기준, 소재지(본점/사업장 구분),
기수혜 제외, 영업상태. **`확인됨`은 "공고상 신청자격 확인"이지 선정 가능성·대출 심사 통과 예측이 아니며**,
보고서에 이 구분을 명시한다.

**원문 검증 규칙**:
- 후보는 반드시 상세공고 원문에서 검증한다. 자격요건이 첨부파일에만 있는 경우가 흔하므로
  첨부 PDF/HWPX는 텍스트 추출해 읽는다. HWP(구버전 바이너리)는 추출 실패 시 명시적으로 실패 처리한다
- **첨부를 읽지 못한 후보는 `확인됨` 판정 금지** → `확인 필요` + "첨부 미확인" 사유
- 각 판정에는 근거 출처(문서·해당 문구)를 기록한다
- 공고 텍스트에 없는 것은 '불명' — 접수기관 확인을 권고한다. 연락처는
  `phone | email | official_contact_url | unavailable` 구조로 (전화번호가 없는 공고도 있다)

### 3단계 — 보고서

구조 (위에서 아래로):

1. **오늘 신청할 것** — 전 유형 통합 `확인됨` 목록, 마감 D-day 오름차순. 스킬의 핵심 약속이므로 최상단
2. **coverage_manifest** — 소스별 성공/실패/수동확인, 수집 건수, 미커버 영역 한계 고지
3. **유형별 상세** — 💰/🏦/🎓 순, 각 유형 안에서 판정 상태순. 🏦에는 "갚아야 하는 돈" 문구와
   한도·금리(기준일)·기간·취급기관 표기
4. **사업전환 후보** — 해당 시에만, 별도 섹션

톤은 사장님이 읽는 문서 — 관공서 용어에 괄호 해설.

### diff 모드 (재조사)

프로필의 `last_survey_dir`가 있으면 증분 조사. **마감일만 비교하지 않는다**:

- 변경 감지 대상: 제목, status, 접수기간, 금리, 한도, 기관, `content_hash`(정규화 상세본문+첨부 해시)
- 정정·변경·조기마감 공고는 원 공고와 연결해 보고
- **정책자금 전체와 직전 `확인됨` 항목은 diff 결과와 무관하게 매 조사마다 접수상태(예산소진 포함)를 재검증**
- 재크롤 실패 소스는 "미갱신"으로 coverage_manifest에 명시
- 보고서 구조: 신규 / 변경(마감·조건) / 소멸된 확인됨 항목(기회 소멸 알림) / 승계 요약

## 4. 소스 계약

| 소스 | 지위 | 어댑터 | 계약 |
|---|---|---|---|
| 소상공인24 (sbiz24.kr) | **필수** | `sbiz_crawl.py` (신규) | SPA + JSON API (`/api/combinePbanc/list` 확인됨, 파라미터 스파이크 필요). 모집중 공고 전수 + 상세 + 첨부 URL |
| 기업마당 (bizinfo) | **필수** | `sources_crawl.py` (이식·개조) | **소상공인 분야 필터 + 전 페이지 순회로 개조** (원본은 최근 30페이지만 — 전수 아님). totalCount 대비 수집률 검증 |
| 소진공 정책자금 (ols.semas.or.kr / semas.or.kr) | **필수** (loan이 핵심 범위이므로 수동 폴백은 계약 위반) | `semas_loan_crawl.py` (신규) | 연간 융자계획·변경공고 + 회차별 접수상태(예산소진 포함). 직접/대리대출 구분 |
| 지역신용보증재단 | **필수(지역 레지스트리)** | `references/region-registry.md` + 어댑터 또는 구조화된 수동절차 | 프로필 지역의 재단 + 지자체 협약보증. 17개 재단 전체 크롤러는 유지비 과다 → 사용자 지역 재단만 대상으로 하고, 자동화 불가 시 **수동확인 절차 + coverage_manifest에 manual 표기** |
| 지자체·경제진흥원 포털 | 조건부 (지역 레지스트리) | region-registry 기반 | 기업마당이 지자체 공고를 전부 커버하지 못함(ir-search 운영 경험과 일치). 주요 광역 단위부터 등록, 미등록 지역은 한계 고지 |

**크롤러 스파이크 완료 조건** (구현 1순위, 소스별로 통과해야 어댑터 구현 착수):
API/페이지 구조 발견 + 이용조건·robots 확인 + 호출속도 상한 결정 + 인증 필요성 + 총건수 확인 방법 +
상세·첨부 접근 방법 + status 필드 존재 + 안정 ID 존재.

**중단선**: 403·CAPTCHA·로그인 요구를 만나면 TLS 지문 우회나 내부 API 반복 호출로 뚫지 않는다 —
공식 API·RSS·수동확인 절차로 전환하고 coverage_manifest에 기록한다.

## 6. 테스트

- **파서 골든 테스트**: 저장된 HTML/JSON fixture 기반 — 사이트 변경과 무관하게 재현
- **크롤러 계약 테스트**: 페이지네이션 종료·중복·0건·HTTP 200 차단페이지 감지, totalCount 대비 수집률 ≥ 95%
- **첨부 테스트**: 자격요건이 첨부에만 있는 공고 fixture — 첨부 미확인 시 `확인됨` 금지 검증
- **판정 경계값 테스트**: 상시근로자 4/5명·9/10명, 업력·연령 경계, 매출 구간 경계, '불명'이 `확인됨`으로 승격되지 않음
- **diff 테스트**: 금리·status만 변경된 정책자금이 변경으로 감지되는지
- **트리거 evals**: sole-search 트리거 시나리오 + ir-search로 가야 할 문구의 네거티브 케이스 + **두 스킬 동시 설치 상태의 충돌 케이스**
- **E2E 4종**: 전국형 보조금 / 지역형 / 정책자금형 / 명백한 부적격형 프로필 각 1회 전체 파이프라인

## 7. 비범위 (YAGNI)

- 사업자등록증 이미지 인식 (후속 후보)
- 신청서 작성 대행 — support-document-writer 스킬의 영역
- 창업지원·R&D·입주공간 — ir-search의 영역
- 다중 고객 프로필 관리 (대행사 모드)
- 상시 제도 해설 (세율·4대보험 요율 등 공고가 아닌 것)
- 17개 지역신보 전체 자동 크롤링 (사용자 지역만)
